> 작업 목록은 PLAN.md, 작업 규칙은 CLAUDE.md 참조.

## 1. 레퍼런스 (북극성)

- **Cookie Clicker** — 장르의 원형이자 수치 구조의 교과서

## 2. 게임 디자인

### 2.1 코어 루프

솥 클릭 → 마나 획득 → 생산 시설 구매 → 초당 마나(MPS) 상승 → 더 비싼 시설/업그레이드 → 숫자가 감당 안 되면 **각성(프레스티지)** → 영구 배율 얻고 처음부터, 더 빠르게.

### 2.2 자원과 클릭

- 기본 자원: **마나**. 클릭당 1 (업그레이드로 성장)
- 표시 포맷: 1,234 → 1.23K → 1.23M → 1.23B → 1.23T → 1.23aa … (포맷터 유틸 필수)

### 2.3 생산 시설 6티어 (data/generators.ts — 출발 수치)

| 티어 | 이름 | 기본 비용 | 생산(MPS) |
|---|---|---|---|
| T1 | 견습생 | 15 | 0.1 |
| T2 | 마법 솥 | 100 | 1 |
| T3 | 허브 정원 | 1,100 | 10 |
| T4 | 마법진 | 10,000 | 75 |
| T5 | 정령 계약 | 130,000 | 640 |
| T6 | 드래곤 둥지 | 1,400,000 | 8,500 |

> T8.1 밸런싱 반영(시뮬레이션 검증 완료): 6티어 구성은 업그레이드 밀도가 낮아
> 원안(47/260/1,400)의 회수기간 곡선(150s→1,000s)으로는 후반 성장이 정체됨.
> T3~T6 상향으로 §2.8 목표표 4개 지점 모두 오차 ±30% 이내 달성
> (T2 1.00 / T4 1.21 / T6 1.03 / 첫 각성 1.26 — 활성 플레이 시뮬 기준).

- 구매 비용 성장: `비용 = 기본비용 × 1.15^보유수` (장르 표준 상수)
- 구매 버튼: ×1 / ×10 / ×MAX 토글

### 2.4 업그레이드 (data/upgrades.ts)

- 티어별 보유 수 마일스톤(10/25/40/50개)에서 해금: 해당 티어 생산 ×2
- 클릭 강화: "클릭 = MPS의 1%" 계열 (후반에도 클릭이 죽지 않게)
- 시너지: "견습생 1명당 솥 생산 +1%" 같은 교차 배율 2~3개 (재미 양념)

> **D-3.4 밸런스 조정(data)**:
> - 마일스톤 40단계 추가(costMult 200, ×2) — 50단계 앞 갭(40~55분 정체) 완화.
> - "마나 공명 II"(click-mps-2): 500K→80K, +1%p→+2%p(합계 3%) — 가치 함정 제거.
> - 소형 초반 업그레이드 2종: "견습생의 열정"(견습생5, 300, ×2), "솥 예열"(솥3, 800, ×1.5) — 5~15분 공백 해소.
> - 재검증(활성 플레이 그리디 시뮬, 목표표 대비): 데이터 변경 전후 마일스톤 이동 <4%(차분 비교) — 기존 ±30% 밸런스 유지.

### 2.5 각성 (프레스티지)

- 조건: 누적 마나 1e9 이상
- 획득: `스타더스트 = floor( sqrt(누적 마나 / 1e9) )`
- 효과: 스타더스트 1개당 전체 생산 **+10% (영구)**
- 리셋 범위: 마나/시설/업그레이드 초기화. 유지: 스타더스트, 스타더스트 상점, 업적, 통계
- 각성 확인 모달에 "지금 각성하면 +N 스타더스트" 미리보기 필수 (안 그러면 유저가 결정을 못 함)

> **D-3 각성 리워크(밸런스·시스템)**:
> - 스타더스트 상점(data/stardustShop.ts): 각성 화폐 소비처. 비용 = baseCost × 2^레벨.
>   견습 마법사단(시작 견습생 5×Lv) · 공명 증폭(클릭 +1%p/Lv) · 꿈꾸는 솥(오프라인 효율 50→75%) · 시간의 모래(오프라인 캡 8→12h).
> - 첫 각성 체감 보정: 첫 각성(totalPrestiges=0) 한정 스타더스트 +2(FIRST_PRESTIGE_BONUS) — "같은 인생 재방송" 완화(U7).
> - "다음 ✨+1까지 (n+1)²×1e9" 상시 표시 + 임계 도달 후에도 누적 진행 유지.

> **E-2 프레스티지 깊이(스타더스트 상점 2단·챌린지 런·후반 곡선 검토)**:
> - **E-2.1 상점 2단 확장(data/stardustShop.ts)**: 기본 4종에 추가 —
>   · 생산 트리 "별의 축복: {티어명}" 8종(티어별, GENERATORS 파생): 해당 티어 생산 ×1.5/Lv(곱), maxLevel 3,
>     baseCost 티어순 2/3/4/6/8/12/20/30(곡선 ×2^Lv). 효과 kind `generatorMult`+generatorId(무결성 테스트 자동 검증).
>     티어 배율은 totalMps가 stardustUpgrades로 반영해 개당 실효값·구매 델타·헤더 MPS가 일관된다.
>   · 자동화 "공방 관리인"(effect `automation`, maxLevel 3, 비용 10/20/40): 오프라인 정산 직후 `applyOfflineAutomation()`이
>     로드 시점 마나로 1회 그리디 자동 구매 — Lv1 시설 배율 업그레이드, Lv2 +클릭/시너지, Lv3 +시설 그리디.
>   · 상점 모달은 섹션(기본/생산/자동화) 구분 + wide.
> - **E-2.2 챌린지 런(세이브 v9, data/challenges.ts)**: 각성과 함께 제약을 걸고 성공 시 영구 생산 배율 —
>   · 침묵의 손🙅(클릭 무효)·금욕의 공방🚫(업그레이드 금지) → 각 +25%(그 생을 제약 지키며 각성 = 성공, prestige에서 판정).
>   · 시간의 시험⏱️(2시간 내 각성 조건 달성) → +50%(임계 도달 시점 성공/실패 판정, withAchievements).
>   · 상태: activeChallenge/completedChallenges(v9, migrate v8→v9). 완료 보상은 `challengeMultiplier`로 composeGlobalMult에 합류.
>   · 시작: 각성 확인 모달의 "챌린지와 함께 각성" 섹션. 진행 중 배지+포기 버튼(보상 없이 해제). 완료는 ✅ 재선택 불가.
> - **E-2.3 후반 곡선 검토(시뮬)**: 8시간 활성 그리디+각성 반복 시뮬(각성=누적 스타더스트 2배 시점, 상점 그리디).
>   실측: 8h에 각성 3회·누적 스타더스트 12·티어 8 도달(마지막 생)·업적 17/40. 티어 첫 구매 T2 2.4분/T4 25.8분/T6 76분,
>   첫 각성 ~2h로 §2.8 목표표와 일치 — 신규 상점 데이터가 코어 곡선을 왜곡하지 않음(성장 정지·폭주 없음). 스타더스트 50+는
>   다회 세션 장기 목표(sqrt 각성식 특성). 추가 데이터 수치 조정 불필요.

### 2.6 오프라인 수익

- 접속 시: `경과시간(최대 8h) × MPS × 50%` 지급
- "돌아온 걸 환영해요! 자리 비운 동안 +X 마나" 팝업 — 방치형 도파민의 핵심 장치

> **D-3.1**: 오프라인 효율·캡은 스타더스트 상점 강화를 반영한 실효값(effectiveOfflineEfficiency/CapMs)을 쓴다.
> 기본 50%/8h → 상점 만렙 시 75%/12h. 로드 경로(autosave)와 tick catch-up 초과분 모두 동일 실효값 적용.

### 2.7 업적 & 주스

- 업적 20개: 클릭 100회, 견습생 50명, 첫 각성, 누적 1e12 … (달성 시 토스트 + 각 +1% 생산 보너스)
- 주스: 클릭 시 +N 숫자 팝, 구매 가능 버튼 하이라이트(살 수 있는 게 생기는 순간의 반짝임), 마일스톤 달성 이펙트, 은은한 배경 앰비언트

> **E-1.3 업적 40개 확장(밸런스 영향)**: 업적 20 → 40(숨겨진 4 포함)으로 상한 보너스가 +20% → **최대 +40% 생산**으로 커진다.
> 이 배율은 스타더스트 배율과 곱으로 합성(recalcDerived)되므로 후반 구간의 성장 바닥을 약간 올린다 —
> 다만 상위 업적(누적 1e18~1e21·MPS 1e10·각성 25회 등)은 사실상 T7~T8/다회 각성 이후에나 달성되어,
> §2.8 초반 목표표(~2시간까지)에는 영향이 없다. 후반 밸런스는 E-1 완료 후 일괄 시뮬로 재검증한다.

## 3. 기술 스택 & 결정 이유

**Vite + React + TypeScript + Zustand**

- 왜 React(캔버스 아님): 방치형은 버튼·리스트·탭·숫자의 게임 = 사실상 반응형 대시보드. React 학습에 최적의 소재
- 왜 Zustand: 전역 게임 상태 + 초당 다회 갱신에는 Context/useReducer가 금방 고통스러워짐. Zustand는 API가 작아 AI도 사람도 실수가 적고, **셀렉터로 리렌더 최적화**를 배우기 좋음
- 게임 tick: `setInterval 100ms`로 `lastTick` 대비 경과시간만큼 생산 반영 — **탭이 백그라운드로 가면 인터벌이 스로틀링되므로, 진실은 항상 타임스탬프 계산** (visibilitychange 복귀 시 catch-up). 이 원칙이 곧 오프라인 수익 로직과 동일 코드
- 저장: localStorage 10초 자동저장 + 수동 저장 + **export/import(Base64 문자열)** — 방치형 전통이자 직렬화 학습
- 세이브 스키마에 `version` 필드 + `migrate(old)` 함수 자리 — 업데이트해도 유저 세이브가 안 깨지게
- 큰 수: v1은 JS number(1e308까지)로 충분. break_eternity.js는 주차장
- 배포: Vercel / GitHub Pages

## 4. 아키텍처

```
src/
  main.tsx / App.tsx
  store/gameStore.ts     # Zustand: 상태 + 액션(click, buy, prestige, tick)
  engine/tick.ts         # 인터벌 + visibilitychange catch-up
  engine/formulas.ts     # 비용/생산/각성 수식 (순수 함수 — 테스트 가능)
  engine/save.ts         # 직렬화, version, migrate, export/import
  engine/offline.ts      # 오프라인 수익 계산
  data/generators.ts, upgrades.ts, achievements.ts
  components/ClickerPanel.tsx, GeneratorList.tsx, GeneratorRow.tsx,
             UpgradePanel.tsx, PrestigeModal.tsx, OfflineModal.tsx,
             AchievementToast.tsx, StatsPanel.tsx, Header.tsx
  utils/format.ts        # K/M/B/T/aa 포맷터
  debug/cheats.ts        # 마나 ×1000, 시간 +1h 시뮬, 리셋
```

핵심 패턴:

- 수식은 전부 `engine/formulas.ts` 순수 함수로 — 유일하게 **단위 테스트를 붙이는 프로젝트** (비용 곡선, 각성 수식은 버그 나면 세이브가 오염됨)
- 컴포넌트는 `useGameStore(selector)`로 필요한 값만 구독. 스토어 통째 구독 금지
- `GeneratorRow`는 memo — 초당 10회 tick에도 값 안 변한 행은 리렌더 안 되는 걸 React DevTools로 직접 확인 (학습 포인트)
- 상태 변형은 스토어 액션에서만. 컴포넌트에서 직접 계산·변형 금지

## 5. 함정 리스트

1. **스토어 통째 구독** — 모든 컴포넌트가 tick마다 리렌더 → 앱 전체가 버벅. 셀렉터 강제 (CLAUDE.md 명시)
2. **인터벌을 진실로 착각** — 백그라운드 탭에서 인터벌은 느려짐. 진실은 `now - lastTick`
3. **세이브 스키마 무버전** — 필드 하나 바꾸면 기존 유저 세이브 전멸. version+migrate는 M4부터
4. **각성 수식 인플레** — 지수 위에 지수를 얹으면 밸런스 즉사. 수식 변경은 formulas 테스트 통과 후에만
5. **부동소수 함정** — 0.1+0.2 문제. 표시는 포맷터로, 비교는 여유값(epsilon)으로
6. **콘텐츠 욕심** — 티어 12개, 자원 3종 같은 확장은 주차장. 6티어로 루프 완성이 먼저
